Introdução sobre Servidor Central e sua estrutura

Este é o mais simples dos algoritmos para se obter exclusão mútua em um sistema distríbuido, pois ele utiliza um servidor central para gerenciar os acessos à seção critica. Todo processo P que quiser acessar a sessão crítica deverá solicitar um ao servidor central. O gerenciamento desse token é feito pelo servidor central através de troca de mensagens com os processos. A troca de mensagens funciona da seguinte forma: processos enviam mensagens para o servidor central e aguardam a resposta dele. As mensagens tem em sua composição um identificador do processo emisor (Pi) e o conteudo da mensagem, que pode ser: solicitação ou liberação acesso. Todo processo que receber resposta do servidor central, tem liberação para acessar a sessão crítica. É necessário uma administração dessas respostas para garantir a exclusão mútua. Para isso, existe uma fila para armazenar as requisições dos processos que utiliza a lógica do "Primeiro a Entrar é o Primeiro a Sair" (FIFO - First in First out).

*Titulos se necessário*

Seja P, um conjunto de processos Pi, com 0 ≤ i < n, onde n é o número de processos participantes da computação, cada processo Pi que deseja entrar na seção critica deve obter um token de acesso. Existe somente um token para ser distribuído entre os processos requisitantes.

O gerenciamento de acesso da sessão crítica pelo servidor funciona da seguinte forma:

  • Se o servidor estiver com o token, ele responde a primeira solicitação que estiver na sua fila de mensagens.
  • Se o servidor não estiver com o token ele não responderá e irá enfileirar as solicitações na fila de mensagem.
  • O servidor recebe a ficha de volta quando um processo envia uma mensagem de liberação de acesso

Desta forma este algoritmo satisfaz as propriedades M1, M2 e M3 para a exclusão mútua. Um ponto negativo desse dele é o atraso na resposta de mensagens, pois cada processo pode ficar indeterminadamente com o token e assim deixar as requisições muito tempo em espera.
