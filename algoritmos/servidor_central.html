Introdução sobre Servidor Central e sua estrutura

Em um sistema distríbuido é necessário efetuar a distribuição e coordenação de acesso à sessão crítica para evitar inconsistência de dados, erros de processamento e bloqueio de recursos indeterminadamente. Este é o mais simples dos algoritmos para se obter exclusão mútua em um sistema distríbuido, pois ele utiliza um Servidor Central para gerenciar e coordenar os acessos à seção critica. Todo processo P que quiser acessar a sessão crítica deverá solicitar um servidor central token ao servidor central. O gerenciamento desse token é feito pelo servidor central através de troca de mensagens com os processos.

A troca de mensagens é centralizada no servidor, ou seja, ele recebe e responde mensagem de todos os processos, enquanto que os processos não se comunicam entre si. A dinâmica das mensagens funciona da seguinte forma:

  • Processos enviam mensagens para o e aguardam a resposta dele.
  • As mensagens tem em sua composição um emisor (Pi) e o conteudo da mensagem, que pode ser: solicitação ou liberação da sessão crítica.

Todo processo que receber resposta do servidor central, receber o , tem automaticamente liberação para acessar a sessão crítica. É necessário uma administração dessas respostas para garantir a exclusão mútua. Para isso, o servidor central tem uma para armazenar as requisições dos processos que utiliza a lógica do "Primeiro a Entrar é o Primeiro a Sair" (FIFO - First in First out).

A estrutura lógica desse modelo é composta pelo servidor central e seus canais de comunicação com cada processo. Este modelo não tem uma topologia definida, mas assemelha-se com a topologia em Estrela.

Seja P, um conjunto de processos Pi, com 0 ≤ i < n, onde n é o número de processos participantes da computação, cada processo Pi que deseja entrar na seção critica deve obter um token de acesso. Existe somente um token para ser distribuído entre os processos requisitantes e seu gerenciamento pelo servidor central funciona da seguinte forma:

  • Se o servidor estiver com o token, ele responde a primeira solicitação que estiver na sua fila de mensagens.
  • Se o servidor não estiver com o token ele não responderá e irá enfileirar as solicitações na fila de mensagem.
  • O servidor recebe a ficha de volta quando um processo envia uma mensagem de liberação de acesso

Desta forma este algoritmo satisfaz as propriedades para a exclusão mútua. Porém, o ponto negativo é o atraso na resposta de mensagens, pois cada processo pode ficar indeterminadamente com o token e assim deixar as requisições muito tempo em espera.

Abaixo foi construído um modelo para exemplificar o funcionamento do algoritmo. Na barra horizontal você pode selecionar de 3 a 8 processos para participarem da computação. Para iniciar a simulação aperte PLAY, para avançar aperte AVANÇAR e para reiniciar aperte RESET.

Ao clicar em RESET, o algoritmo será iniciado e será gerado uma fila aleatória, que pode conter de 1 a 8 processos.

No modelo seguinte, experimente modificar o algoritmo:

  • Defina o número de processos através da barra horizontal.
  • Apague a fila de processos caso queira fazer uma fila personalizada, senão aperte em RESET para gerar uma fila.
  • Selecione o processo no campo de seleção.
  • Adicione ou remova processos selecionados da fila.
  • Altere a posição do token para o processo selecionado.
